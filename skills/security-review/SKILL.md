# security-review

OWASP・CWE・バグバウンティ・インシデント対応の視点からコードを体系的にレビューし、
悪用可能な脆弱性を発見して重大度付きで報告する。

## Step 1. スコープと対象コードの取得

`$ARGUMENTS` に応じて対象を決定する:

- PR 番号/URL → `gh pr diff <number>` で差分を取得、`gh pr view <number>` で背景を確認
- ファイルパス → Read/Grep で対象ファイルを読み込む
- 指定なし → `git diff origin/main...HEAD` で現ブランチの変更を取得

対象が確定したら、以下を並列で把握する:
- 変更されたファイルの種類（認証・API・DB・設定・フロントエンド等）
- 外部入力の受け取り箇所（HTTP パラメータ・ファイル・環境変数・IPC）
- データストアへのアクセス箇所（DB・ファイル・キャッシュ・外部 API）

## Step 2. 攻撃面マッピング（バグバウンティ手法）

ホワイトハッカーが最初に行う偵察ステップ。
**信頼境界**（ユーザー入力がシステム内部に到達する箇所）を列挙し、
各境界でどの検証・エスケープ・認可チェックが行われているかを追跡する。

データフローを「ソース → プロセス → シンク」でトレースし、
バリデーションが欠落・不完全な経路を特定する。

## Step 3. OWASP Top 10 による体系的チェック

`references/owasp-top10.md` を読み込み、各カテゴリを対象コードに適用する。

チェック時の優先度（インシデント統計に基づく）:
1. アクセス制御の欠陥（全侵害の最多原因）
2. 認証・セッション管理の不備
3. インジェクション（SQL・コマンド・LDAP・XSS）
4. 設定ミス・デフォルト認証情報
5. ソフトウェアサプライチェーンのリスク
6. 暗号化の欠陥（弱いアルゴリズム・鍵管理）
7. 不適切なエラー処理・ログ記録

## Step 4. シークレット・認証情報スキャン

以下のパターンを Grep で検索する（全ファイル対象）:

```bash
# 高エントロピー文字列・APIキー・トークンのパターン
grep -rn --include="*.{js,ts,py,go,java,rb,php,env,yaml,yml,json,tf}" \
  -E "(password|passwd|secret|api_key|apikey|token|credential|private_key)\s*[:=]\s*['\"][^'\"]{8,}" .

# AWS・GCP・Azure 認証情報
grep -rn -E "(AKIA[0-9A-Z]{16}|AIza[0-9A-Za-z_-]{35})" .

# .env・設定ファイルのコミット漏れ
git diff --name-only | grep -E "\.(env|pem|p12|pfx|key|crt)$"
```

**重要**: バグバウンティでの頻出発見は「テストコード中のハードコード認証情報」。
テストファイルも含めて検索すること。

## Step 5. 依存関係の脆弱性監査（サプライチェーン）

インシデントの 28% がサプライチェーン由来。以下を確認する:

```bash
# Node.js
npm audit --json 2>/dev/null | head -100

# Python
pip-audit 2>/dev/null || safety check 2>/dev/null

# Go
govulncheck ./... 2>/dev/null

# 汎用: OSV-Scanner
osv-scanner . 2>/dev/null
```

コマンドが存在しない場合は `package.json`・`requirements.txt`・`go.mod` を読み込み、
主要な依存関係を確認する。

## Step 6. 脆弱性パターンの詳細検証

発見した疑わしい箇所について `references/vulnerability-patterns.md` を参照し、
実際に悪用可能かを検証する。

バグバウンティの鉄則: **複数の低深刻度の問題を連鎖させて高深刻度にできないか**を常に検討する。

## Step 7. 報告書の出力

### 重大度分類（CVSS ベース）

| 重大度 | CVSS スコア | 基準 |
|--------|------------|------|
| Critical | 9.0–10.0 | 認証不要・リモートから即座に悪用可能・データ損失・RCE |
| High | 7.0–8.9 | 認証済みユーザーが特権昇格・機密データ漏洩 |
| Medium | 4.0–6.9 | 条件付き悪用・部分的な情報漏洩 |
| Low | 0.1–3.9 | 実環境での悪用困難・情報収集のみ |
| Info | — | セキュリティ上の改善推奨（脆弱性ではない） |

### 出力フォーマット

```
## セキュリティレビュー結果

### サマリー
- Critical: N 件 / High: N 件 / Medium: N 件 / Low: N 件

### 発見事項

#### [CRITICAL/HIGH/MEDIUM/LOW] <タイトル>
- **場所**: `path/to/file.ts:行番号`
- **カテゴリ**: OWASP A0X / CWE-XXX
- **説明**: 何が問題か（1〜2 文）
- **攻撃シナリオ**: 攻撃者がどう悪用するか（バグバウンティ視点）
- **修正方法**: 具体的なコード例または修正指針
- **参考**: [OWASP Cheat Sheet](url) / CVE-XXXX-XXXXX

### 対応優先度
Critical・High は即座にマージをブロックし修正する。
Medium は今スプリント内に対応する。Low・Info は技術的負債として追跡する。
```

### 問題が見つからない場合

「問題なし」の場合も確認したカテゴリを明示し、
何を根拠に安全と判断したかを記載する（証拠なき "クリーン" 宣言は不可）。
