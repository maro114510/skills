# pr-review-pe

PR の diff を読み、PE が本当に気にすることだけを指摘する。速さと的確さを両立させるため、
6つの PE 観点に絞ってレビューし、細かい nits は原則スキップする。

## Step 1: PR 特定

`$ARGUMENTS` を解析する:

- 数値のみ（例: `123`）→ そのまま PR 番号として使用
- GitHub URL（`/pull/123` を含む）→ URL から番号を抽出
- 引数なし → 現在のブランチの PR を `gh pr view --json number` で取得

PR が特定できない場合は次のように伝えて停止する:
「現在のブランチ `<branch名>` に PR が見つかりません。PR 番号または URL を引数で指定してください（例: `9` / `https://github.com/org/repo/pull/9`）。」
（gh コマンドが一時障害で失敗した場合も同様に停止し、ユーザーに手動でPR番号を指定するよう案内する。）

## Step 2: データ収集（並列実行）

```bash
# PR メタ情報
gh pr view $PR_NUMBER --json title,body,author,baseRefName,headRefName,additions,deletions,changedFiles

# 差分（ハルシネーション防止の基盤）
gh pr diff $PR_NUMBER

# 既存コメント・レビュー（重複防止）
gh pr view $PR_NUMBER --json comments,reviews
```

## Step 3: PE 観点でレビュー

diff を読み、以下の6観点で問題を探す。**各指摘は必ず diff の該当コードを引用してから行う**
（引用できない = diff に存在しない = 指摘してはならない）。

既存コメントで既に指摘されている内容は重複して指摘しない。
PR description でユーザー自身が説明・対処済みの問題も指摘しない。

### PE が重視する6観点

1. **アーキテクチャ**: 責務分離、依存関係の方向、設計パターンの適切さ
2. **スケーラビリティ**: N+1 クエリ、ボトルネック、キャッシュ設計、データ量増加時の挙動
3. **セキュリティ**: 入力検証、認証・認可の欠如、機密情報の扱い、インジェクション等
4. **API 設計**: 後方互換性の破壊、インターフェースの明確さ、バージョニング
5. **エラーハンドリング**: 障害時の挙動、エラーの握りつぶし、リトライ設計、タイムアウト
6. **運用性**: ログ・メトリクスの適切さ、デプロイ影響範囲、設定の外部化

### 重要度の判断基準

| タグ | 基準 | 対応 |
|------|------|------|
| [must] | 本番で問題が起きる / API 互換性を破壊 / セキュリティリスク | 修正必須 |
| [imo] | 改善が望ましいがブロッカーではない | 修正任意 |
| [nits] | 原則スキップ — PE 観点では命名・スタイルの細かい指摘は行わない | — |
| [ask] | 設計判断に影響する意図が PR description から読めない場合のみ | 回答必須 |
| [good] | 特に優れた設計判断 | — |
| [next] | 今回スコープ外の将来検討すべき設計課題 | 修正不要 |

## Step 4: 自己検証

出力前に以下を確認し、該当する指摘は削除する:

- [ ] Step 2 で取得した diff の実際のコードを参照しているか？
- [ ] PR description で既に説明・対処されていないか？
- [ ] 既存コメント・レビューと重複していないか？
- [ ] severity（[must]/[imo]）は変更の影響規模と一致しているか？

## Step 5: 出力フォーマット

```
## PR #NNN レビュー: <タイトル>

### 総評
（2-3文。PE 視点での全体印象と最重要懸念）

### 指摘事項

**[must] `path/to/file:42`**: 問題の説明
> `diff から引用したコード`
→ 改善案: 具体的なコードまたは手順

**[imo] `path/to/file:87`**: 問題の説明
> `diff から引用したコード`
→ 改善案: ...

### 良い点
- [good] `path/to/file:15`: 評価できる設計判断

### 自己検証結果
全 N 件の指摘を確認。除外: N 件（根拠なし / PR description 説明済み / 既存コメント重複）
```

問題がなければ `### 指摘事項` セクション内に「指摘事項なし。Approve 推奨。」と書く（見出しは省略しない）。
良い点がある場合は `### 良い点` セクションを含める（指摘事項なしでも出力してよい）。`[good]` タグは `### 良い点` セクション専用。
`### 自己検証結果` の除外件数が 0 の場合は「除外: 0 件」と記載する。
